## 1. 设计思路

### 1.1 总体设计原则

根据阿里巴巴开发手册和需求文档，本数据库设计遵循以下原则：

1. **数据分层存储**：将热点数据和冷数据分离，热点数据使用Redis缓存，冷数据存储在MySQL中
2. **合理使用索引**：为查询频繁的字段建立合适的索引，提高查询效率
3. **字段命名规范**：采用下划线命名法，字段名清晰明了
4. **适当冗余设计**：在必要时进行适当的数据冗余，减少多表关联查询
5. **安全性考虑**：敏感信息（如密码）进行加密存储
6. **避免大字段**：不在主表中存储大文本或二进制数据

### 1.2 MySQL与Redis职责划分

#### 1.2.1 MySQL职责

- 用户基础信息的持久化存储
- 用户积分和游戏统计数据的永久记录
- 积分流水详情的历史记录存储
- 游戏房间的基础信息存储

#### 1.2.2 Redis职责

- 用户在线状态的实时维护
- 匹配队列的临时存储和管理
- 游戏房间实时状态的缓存
- 用户会话信息的临时存储
- 棋盘状态的实时缓存

## 2. MySQL数据库设计

### 2.1 数据表设计

#### 2.1.1 用户表（users）

存储用户基础信息和游戏统计数据

| 字段名 | 数据类型 | 长度 | 允许空 | 默认值 | 主键 | 说明 |
| ------ | -------- | ---- | ------ | ------ | ---- | ---- |
| id | BIGINT | 20 | 否 | 无 | 是 | 用户唯一标识 |
| username | VARCHAR | 50 | 否 | 无 | 否 | 用户账号，唯一 |
| password | VARCHAR | 255 | 否 | 无 | 否 | 用户密码（加密存储） |
| current_score | INT | 11 | 否 | 0 | 否 | 当前积分 |
| total_games | INT | 11 | 否 | 0 | 否 | 总比赛场数 |
| win_games | INT | 11 | 否 | 0 | 否 | 获胜场数 |
| created_at | TIMESTAMP | 0 | 否 | CURRENT_TIMESTAMP | 否 | 创建时间 |
| updated_at | TIMESTAMP | 0 | 否 | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 否 | 更新时间 |

```sql
CREATE TABLE `users` (
  `id` BIGINT(20) NOT NULL AUTO_INCREMENT COMMENT '用户唯一标识',
  `username` VARCHAR(50) NOT NULL COMMENT '用户账号',
  `password` VARCHAR(255) NOT NULL COMMENT '用户密码（加密存储）',
  `current_score` INT(11) NOT NULL DEFAULT 0 COMMENT '当前积分',
  `total_games` INT(11) NOT NULL DEFAULT 0 COMMENT '总比赛场数',
  `win_games` INT(11) NOT NULL DEFAULT 0 COMMENT '获胜场数',
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_username` (`username`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户基础信息表';
```

#### 2.1.2 积分流水表（score_records）

存储用户积分变化的详细记录

| 字段名 | 数据类型 | 长度 | 允许空 | 默认值 | 主键 | 说明 |
| ------ | -------- | ---- | ------ | ------ | ---- | ---- |
| id | BIGINT | 20 | 否 | 无 | 是 | 记录唯一标识 |
| user_id | BIGINT | 20 | 否 | 无 | 否 | 用户ID |
| opponent_id | BIGINT | 20 | 否 | 无 | 否 | 对手ID |
| room_id | BIGINT | 20 | 否 | 无 | 否 | 游戏房间ID |
| game_result | TINYINT | 1 | 否 | 无 | 否 | 对局结果：1胜利，0失败 |
| score_change | INT | 11 | 否 | 无 | 否 | 积分变化：+5或-3 |
| game_time | TIMESTAMP | 0 | 否 | CURRENT_TIMESTAMP | 否 | 对局时间 |
| created_at | TIMESTAMP | 0 | 否 | CURRENT_TIMESTAMP | 否 | 创建时间 |

```sql
CREATE TABLE `score_records` (
  `id` BIGINT(20) NOT NULL AUTO_INCREMENT COMMENT '记录唯一标识',
  `user_id` BIGINT(20) NOT NULL COMMENT '用户ID',
  `opponent_id` BIGINT(20) NOT NULL COMMENT '对手ID',
  `room_id` BIGINT(20) NOT NULL COMMENT '游戏房间ID',
  `game_result` TINYINT(1) NOT NULL COMMENT '对局结果：1胜利，0失败',
  `score_change` INT(11) NOT NULL COMMENT '积分变化：+5或-3',
  `game_time` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '对局时间',
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  KEY `idx_user_id_game_time` (`user_id`, `game_time`),
  KEY `idx_opponent_id` (`opponent_id`),
  KEY `idx_room_id` (`room_id`),
  FOREIGN KEY (`room_id`) REFERENCES `game_rooms`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='积分流水记录表';
```

#### 2.1.3 游戏房间表（game_rooms）

存储游戏房间的基础信息

| 字段名 | 数据类型 | 长度 | 允许空 | 默认值 | 主键 | 说明 |
| ------ | -------- | ---- | ------ | ------ | ---- | ---- |
| id | BIGINT | 20 | 否 | 无 | 是 | 房间唯一标识 |
| black_player_id | BIGINT | 20 | 否 | 无 | 否 | 黑方用户ID |
| white_player_id | BIGINT | 20 | 否 | 无 | 否 | 白方用户ID |
| current_turn | TINYINT | 1 | 否 | 1 | 否 | 当前轮次：1黑方，0白方 |
| game_status | TINYINT | 1 | 否 | 0 | 否 | 游戏状态：0进行中，1黑胜，2白胜，3认输结束 |
| winner_id | BIGINT | 20 | 是 | NULL | 否 | 获胜者ID |
| started_at | TIMESTAMP | 0 | 否 | CURRENT_TIMESTAMP | 否 | 游戏开始时间 |
| ended_at | TIMESTAMP | 0 | 是 | NULL | 否 | 游戏结束时间 |
| created_at | TIMESTAMP | 0 | 否 | CURRENT_TIMESTAMP | 否 | 创建时间 |

```sql
CREATE TABLE `game_rooms` (
  `id` BIGINT(20) NOT NULL AUTO_INCREMENT COMMENT '房间唯一标识',
  `black_player_id` BIGINT(20) NOT NULL COMMENT '黑方用户ID',
  `white_player_id` BIGINT(20) NOT NULL COMMENT '白方用户ID',
  `current_turn` TINYINT(1) NOT NULL DEFAULT 1 COMMENT '当前轮次：1黑方，0白方',
  `game_status` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '游戏状态：0进行中，1黑胜，2白胜，3认输结束',
  `winner_id` BIGINT(20) DEFAULT NULL COMMENT '获胜者ID',
  `started_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '游戏开始时间',
  `ended_at` TIMESTAMP NULL DEFAULT NULL COMMENT '游戏结束时间',
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  KEY `idx_black_player_id` (`black_player_id`),
  KEY `idx_white_player_id` (`white_player_id`),
  KEY `idx_game_status` (`game_status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='游戏房间表';
```

### 2.2 索引设计说明

1. **用户表（users）**
   - 主键索引：id（自带）
   - 唯一索引：uk_username（username）- 确保用户名唯一性，用于登录验证

2. **积分流水表（score_records）**
   - 主键索引：id（自带）
   - 复合索引：idx_user_id_game_time（user_id, game_time）- 用于按用户查询积分流水并按时间排序
   - 普通索引：idx_opponent_id（opponent_id）- 用于查询对手信息
   - 普通索引：idx_room_id（room_id）- 用于查询特定游戏房间的积分记录
   - 外键约束：FK_room_id关联game_rooms表，确保数据完整性

3. **游戏房间表（game_rooms）**
   - 主键索引：id（自带）
   - 普通索引：idx_black_player_id（black_player_id）- 用于查询用户的游戏记录
   - 普通索引：idx_white_player_id（white_player_id）- 用于查询用户的游戏记录
   - 普通索引：idx_game_status（game_status）- 用于查询不同状态的游戏

## 3. Redis数据库设计

### 3.1 键值设计

#### 3.1.1 用户在线状态

```
# 存储用户在线状态和会话信息
key: user:online:{user_id}
value: {
  "username": "用户名",
  "score": 当前积分,
  "connection_id": "连接标识",
  "status": "online/matching/gaming",
  "last_active": "最后活跃时间"
}
```

#### 3.1.2 匹配队列

```
# 存储等待匹配的用户队列
key: match:queue
value: sorted set {
  score: 用户积分,
  member: user_id
}
```

#### 3.1.3 游戏房间状态

```
# 存储游戏房间的实时状态
key: room:{room_id}
value: {
  "black_player_id": 黑方用户ID,
  "white_player_id": 白方用户ID,
  "current_turn": 当前轮次,
  "board_state": "棋盘状态（15x15数组）",
  "game_status": 游戏状态,
  "move_count": 已下棋子数量
}
```

#### 3.1.4 用户会话

```
# 存储用户登录会话信息
key: session:{token}
value: {
  "user_id": 用户ID,
  "username": "用户名",
  "login_time": "登录时间"
}
```

### 3.2 数据结构选择理由

1. **Hash结构**：用于存储用户在线状态、游戏房间状态等复杂结构化数据，便于单字段更新
2. **Sorted Set结构**：用于匹配队列，按积分排序便于实现基于积分的就近匹配算法
3. **String结构**：用于简单的会话token存储，读取效率高

### 3.3 过期策略

1. **用户在线状态**：设置30分钟过期时间，定期心跳刷新
2. **匹配队列**：匹配成功或取消匹配时主动删除，无需设置过期时间
3. **游戏房间状态**：游戏结束后设置1小时过期时间，便于处理异常情况
4. **用户会话**：设置24小时过期时间，支持长时间游戏体验

## 4. 数据一致性保障

### 4.1 MySQL与Redis数据同步策略

1. **用户积分更新**：游戏结束后先更新MySQL数据，再同步更新Redis中的用户在线状态
2. **游戏房间状态**：Redis作为实时状态缓存，游戏结束后将最终结果持久化到MySQL
3. **匹配队列管理**：队列数据仅存储在Redis中，匹配成功后清理队列数据

### 4.2 事务处理

1. **游戏结束积分更新**：使用MySQL事务确保用户积分、游戏统计、积分流水记录的原子性更新
2. **用户注册**：使用事务确保用户信息创建和初始数据设置的一致性
3. **Redis操作**：使用pipeline或lua脚本确保相关操作的原子性

### 4.3 并发控制

1. **积分更新**：使用乐观锁机制，通过version字段或时间戳确保并发更新的正确性
2. **匹配队列**：使用Redis的原子操作确保队列操作的线程安全
3. **游戏状态更新**：通过Redis的single-threaded特性保证游戏状态变更的一致性

## 5. 安全性考虑

### 5.1 密码安全

1. **密码加密**：使用BCrypt等安全的hash算法对用户密码进行加密存储
2. **密码强度验证**：确保密码至少8位，包含字母和数字

### 5.2 数据访问控制

1. **SQL注入防护**：使用参数化查询，避免拼接SQL语句
2. **权限控制**：数据库用户权限最小化，只授予必要的增删改查权限
3. **敏感信息保护**：用户密码等敏感信息加密存储，不在日志中记录

### 5.3 会话安全

1. **Token管理**：使用安全的随机token，设置合理的过期时间
2. **会话验证**：每次请求验证token有效性，防止非法访问
