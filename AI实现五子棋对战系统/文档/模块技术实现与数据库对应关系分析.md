## 1. 业务模块与技术实现对应关系

### 1.1 用户管理模块 (User Module)

#### 1.1.1 功能概述

- 用户注册信息验证和存储（账号唯一性验证、密码安全规则验证）
- 用户基础信息查询和更新
- 用户游戏统计信息管理（总场次、胜场、当前积分）
- 用户数据的CRUD操作
- 用户初始资料创建（初始积分为0，比赛场数为0，获胜场数为0）

#### 1.1.2 技术实现

- **前端技术**：HTML/CSS/JavaScript + DOM操作 + XMLHttpRequest/Fetch API
- **后端技术**：
  - Spring Boot 3.x RESTful API
  - Spring Data JPA + MyBatis-Plus ORM
  - BCrypt密码加密
  - Bean Validation数据验证
  - MySQL事务管理

#### 1.1.3 数据库对应关系

- **MySQL**：
  - `users`表：存储用户基础信息和游戏统计数据
    - 字段：id, username, password, current_score, total_games, win_games, created_at, updated_at
    - 索引：主键索引(id)、唯一索引(uk_username)

- **Redis**：
  - `user:online:{user_id}`：存储用户在线状态和基础信息缓存
    - 数据类型：Hash
    - 字段：username, score, connection_id, status, last_active
    - 过期策略：30分钟过期，心跳刷新

### 1.2 认证授权模块 (Auth Module)

#### 1.2.1 功能概述

- 用户登录验证（账号密码验证）
- 会话管理和Token生成
- 登录状态维护
- 登录错误提示处理（账号不存在、密码错误等明确提示）

#### 1.2.2 技术实现

- **前端技术**：HTML/CSS/JavaScript + 表单验证 + 会话状态管理
- **后端技术**：
  - JWT (JSON Web Token)认证框架
  - Spring Security安全框架
  - BCrypt密码验证
  - Redis会话存储
  - Spring Boot Security配置

#### 1.2.3 数据库对应关系

- **MySQL**：
  - `users`表：验证用户登录凭据
    - 字段：username, password
    - 索引：唯一索引(uk_username)用于登录查询

- **Redis**：
  - `session:{token}`：存储用户登录会话信息
    - 数据类型：String/Hash
    - 字段：user_id, username, login_time
    - 过期策略：24小时过期时间

### 1.3 积分管理模块 (Score Module)

#### 1.3.1 功能概述

- 积分计算逻辑（胜利+5分，失败-3分，最低0分）
- 积分变更记录存储（包含对局时间、对手信息、对局结果、积分变化）
- 积分流水详情查询（按时间顺序排列、分页显示）
- 实时积分更新到用户账户

#### 1.3.2 技术实现

- **前端技术**：HTML/CSS/JavaScript + 分页组件 + 实时积分显示
- **后端技术**：
  - Spring Boot事务管理
  - MyBatis-Plus分页插件
  - Spring Boot Scheduler定时任务
  - Redis缓存更新机制
  - 乐观锁并发控制

#### 1.3.3 数据库对应关系

- **MySQL**：
  - `users`表：更新用户当前积分和游戏统计
    - 字段：current_score, total_games, win_games
    - 索引：主键索引(id)用于积分更新
  - `score_records`表：存储积分变化流水记录
    - 字段：user_id, opponent_id, room_id, game_result, score_change, game_time, created_at
    - 索引：复合索引(idx_user_id_game_time)用于流水查询

- **Redis**：
  - `user:online:{user_id}`：实时更新用户积分缓存
    - 数据类型：Hash
    - 字段：score
    - 过期策略：与用户在线状态同步更新

### 1.4 匹配系统模块 (Matching Module)

#### 1.4.1 功能概述

- 匹配队列管理（加入、退出队列）
- 基于积分的匹配算法实现（就近匹配保证公平性）
- 匹配状态实时更新（显示匹配状态和等待时间）
- 匹配结果通知
- 取消匹配功能处理
- 匹配成功后房间分配
- 黑白方随机分配

#### 1.4.2 技术实现

- **前端技术**：HTML/CSS/JavaScript + WebSocket实时通信 + 状态展示组件
- **后端技术**：
  - Spring Boot WebSocket
  - Redis Sorted Set队列管理
  - Spring Boot Scheduler匹配调度
  - 随机算法实现
  - 多线程匹配处理

#### 1.4.3 数据库对应关系

- **MySQL**：
  - `users`表：获取用户积分用于匹配算法
    - 字段：current_score
    - 索引：主键索引(id)

- **Redis**：
  - `match:queue`：存储等待匹配的用户队列
    - 数据类型：Sorted Set
    - 字段：score(用户积分), member(user_id)
    - 过期策略：匹配成功或取消时主动删除
  - `user:online:{user_id}`：更新用户匹配状态
    - 数据类型：Hash
    - 字段：status(online/matching/gaming)
    - 过期策略：30分钟过期，状态变更时更新

### 1.5 游戏核心模块 (Game Module)

#### 1.5.1 功能概述

- 游戏房间创建和管理
- 棋盘状态维护（15×15棋盘）
- 落子合法性验证
- 五子连珠胜负判定（横、竖、斜方向连续五子检测）
- 游戏回合控制（黑方先行，轮流落子）
- 认输功能处理
- 游戏计时逻辑支持（配合前端计时器）
- 游戏结束状态处理
- 胜负结果通知

#### 1.5.2 技术实现

- **前端技术**：HTML5 Canvas棋盘绘制 + JavaScript游戏逻辑 + WebSocket实时同步
- **后端技术**：
  - Spring Boot WebSocket实时通信
  - 五子棋算法实现（胜负判定算法）
  - Spring Boot定时器（游戏计时支持）
  - 多线程游戏状态管理
  - 游戏状态机设计模式

#### 1.5.3 数据库对应关系

- **MySQL**：
  - `game_rooms`表：存储游戏房间基础信息
    - 字段：id, black_player_id, white_player_id, current_turn, game_status, winner_id, started_at, ended_at, created_at
    - 索引：主键索引(id)、普通索引(idx_black_player_id, idx_white_player_id, idx_game_status)

- **Redis**：
  - `room:{room_id}`：存储游戏房间实时状态
    - 数据类型：Hash
    - 字段：black_player_id, white_player_id, current_turn, board_state, game_status, move_count
    - 过期策略：游戏结束后1小时过期
  - `user:online:{user_id}`：更新用户游戏状态
    - 数据类型：Hash
    - 字段：status(gaming)
    - 过期策略：游戏结束时更新为online状态

### 1.6 实时通信模块 (Communication Module)

#### 1.6.1 功能概述

- WebSocket连接管理
- 实时消息推送（落子、游戏状态变化、匹配状态更新）
- 连接状态监控
- 房间内双方信息同步
- 积分变化实时通知

#### 1.6.2 技术实现

- **前端技术**：WebSocket API + JavaScript事件处理 + 实时UI更新
- **后端技术**：
  - Spring Boot WebSocket框架
  - WebSocket Session管理
  - 消息广播机制
  - 连接心跳检测
  - 异步消息处理

#### 1.6.3 数据库对应关系

- **MySQL**：
  - 无直接数据库操作，主要依赖其他模块提供的数据

- **Redis**：
  - `user:online:{user_id}`：维护用户连接状态
    - 数据类型：Hash
    - 字段：connection_id, last_active
    - 过期策略：30分钟过期，心跳刷新
  - `room:{room_id}`：获取房间状态用于消息推送
    - 数据类型：Hash
    - 字段：用于消息推送的房间状态数据
    - 过期策略：读取操作，不设置过期

## 2. 技术选型合理性

### 2.1 前端技术选型合理性

1. **原生JavaScript + DOM操作**：适合中小型项目，减少框架复杂度，提高页面加载速度
2. **WebSocket**：实现真正的实时双向通信，满足游戏实时性要求
3. **HTML5 Canvas**：适合绘制棋盘和棋子，提供流畅的游戏体验
4. **Fetch API**：现代化的HTTP请求方式，支持Promise，代码简洁

### 2.2 后端技术选型合理性

1. **Spring Boot 3.x + JDK 17**：现代化的Java开发框架，提供完整的企业级功能
2. **MyBatis-Plus**：简化数据库操作，提供丰富的CRUD功能和分页支持
3. **MySQL 8.0**：成熟稳定的关系型数据库，适合存储用户数据和游戏记录
4. **Redis 6.0**：高性能缓存系统，适合存储实时游戏状态和用户会话
5. **JWT认证**：无状态认证方式，适合分布式系统扩展
6. **WebSocket**：服务端实时通信支持，与前端WebSocket配合实现双向通信

### 2.3 数据库设计合理性

1. **MySQL与Redis职责分离**：MySQL负责持久化存储，Redis负责实时数据缓存
2. **合理的索引设计**：基于查询频率设计索引，提高查询效率
3. **适当的数据冗余**：在用户表中存储游戏统计信息，减少复杂查询
4. **事务保证数据一致性**：关键操作使用事务确保数据完整性
5. **Redis数据结构选择**：根据业务场景选择合适的数据结构（Hash、Sorted Set等）

### 2.4 架构设计合理性

1. **模块化设计**：清晰的模块职责划分，便于开发和维护
2. **分层架构**：表现层、业务层、数据层分离，符合软件工程最佳实践
3. **实时通信架构**：WebSocket + Redis结合，提供高效的实时消息推送
4. **缓存策略设计**：多层缓存策略，提高系统性能和用户体验
